

# BRAND-PLAN.md — 가계부 브랜드 시스템 기획

---

## 이 문서의 용도

이 문서는 가계부의 별명(alias) 시스템을 브랜드(brand) 시스템으로 교체하는 장기 작업의 기획서다. 4단계에 걸쳐 진행하며, 각 단계의 작업 세션에서 WORKFLOW.md와 함께 업로드한다.

**WORKFLOW.md와의 관계:**
- WORKFLOW.md에는 브랜드 시스템의 존재와 데이터 스키마만 반영한다 (8번, 9번, 15번)
- 세부 기획, 단계별 작업 순서, Gemini 프롬프트 가이드라인 등은 이 문서를 참조한다
- 4단계 완료 후 이 문서는 아카이브하고, 변경 사항은 WORKFLOW.md에 이미 반영되어 있으므로 기획 문서 없이 유지보수 가능하다

---

## 1. 현재 문제

### 별명 시스템의 한계
- SMS 매출처명이 같은 곳인데 제각각이다 ("씨유홍대3호점", "씨유 인천공항T2교통센터")
- 사용자가 별명/아이콘을 일일이 수동 매핑해야 하고, 새 이름이 나올 때마다 또 설정해야 한다
- 매출처명을 실수로 바꾸면 매핑이 끊어져서 항목이 사라진다
- 별명의 용도가 섞여 있다: 프랜차이즈 묶기(CU)와 카테고리 묶기(외식)를 하나의 필드로 처리
- 카테고리성 별명(외식, 약국)으로 묶으면 그 안에서 상호별 분석이 안 된다

### 대시보드 랭킹의 잡음
- 별명으로 많이 분류했는데도 연간 기타가 54%
- 소액 일회성 매출처가 중요한 매출처를 밀어낸다
- 브랜드 없는 매출처에 기본 고양이 아이콘만 표시되어 구분이 안 된다

---

## 2. 해결 방향

**별명 시스템을 브랜드 시스템으로 교체한다.**

- "별명"이라는 모호한 개념 대신 "브랜드"로 명확히 한다
- 브랜드 인식은 Gemini가 자동으로 한다
- 아이콘은 브랜드당 한 번 사용자가 입력한다
- 소규모 가게는 브랜드 없이 카테고리로 관리한다

**용어 정의:**
- **브랜드 매출처**: Gemini가 프랜차이즈/체인/플랫폼으로 인식한 매출처 (brand 필드에 값이 있음)
- **비브랜드 매출처**: 개인 상호, 동네 가게 등 브랜드로 묶을 수 없는 매출처 (brand가 null)

## 3. 카드 결제 시 (자동 처리)

SMS 수신 → Gemini에게 질의: "이 매출처의 카테고리와 브랜드명은?" → expense에 저장

예시:
- "씨유홍대3호점" → 카테고리: 식비, 브랜드: "CU"
- "나이키의류" → 카테고리: 쇼핑, 브랜드: "나이키"
- "마포진짜원조최대포집" → 카테고리: 외식, 브랜드: null (비브랜드)

Gemini는 카테고리 질의할 때 브랜드명도 함께 판단한다. 이미 API를 호출하는 김에 필드 하나 더 받는 것이라 추가 비용 없음.

"CU" 브랜드에 이미 아이콘이 등록되어 있으면 이 항목에도 자동으로 아이콘이 적용된다. 새로운 브랜드이고 아이콘이 없으면 카테고리 아이콘으로 표시된다.

### 브랜드 결정 우선순위

1. Gemini 호출 (또는 폴백) → category + brand 획득
2. `brandOverrides`에 해당 매출처명이 있으면 → brand를 오버라이드 값으로 교체 (category는 그대로 유지)
3. 없으면 → Gemini/폴백 결과의 brand를 그대로 사용

**중요:** brandOverrides 체크는 Gemini 호출과 무관하게, **category + brand 값이 확정된 후** 최종 brand를 결정하는 시점에서 수행한다. Gemini가 성공하든 폴백이든 상관없이 동일한 시점에서 오버라이드를 적용한다.

### Gemini 응답 형식

```
요청: "씨유홍대3호점" (카드: 삼성카드)
응답: {"category": "food", "brand": "CU"}

요청: "마포진짜원조최대포집" (카드: 신한카드)
응답: {"category": "dining", "brand": null}
```

Gemini가 JSON 형식을 지키지 않을 때의 폴백: JSON 파싱 실패 시 기존처럼 응답 텍스트에서 카테고리 ID를 문자열 매칭하고, brand는 null로 처리한다.

### Gemini 실패 시 폴백 처리

`autoMatchCategoryServer`로 폴백할 때 brand는 항상 null이다. 규칙 기반으로 브랜드를 매칭하지 않는다. 브랜드 로직을 두 곳에서 관리하는 것을 피한다. 폴백으로 저장된 항목은 나중에 `reclassifyAllExpenses`로 브랜드를 채울 수 있다.

**단, 폴백 후에도 brandOverrides 체크는 수행한다.** 폴백 결과의 brand(null)를 오버라이드 값으로 교체할 수 있다.

### 인터페이스 변경

`classifyMerchantWithGemini(merchant, config)` → `classifyMerchantWithGemini(merchant, card, config)`

카드 정보를 함께 넘겨서 백화점 등 카드 기반 힌트를 제공한다.

### Gemini 프롬프트 가이드라인

**브랜드 인식 기준:**
- **브랜드로 인식하는 것:** 전국적 프랜차이즈(CU, 스타벅스, 올리브영), 널리 알려진 상표(나이키, 애플), 체인점(김밥천국, 이삭토스트), 배달앱(배달의민족, 쿠팡이츠, 요기요), 온라인 플랫폼(쿠팡, 네이버, 카카오)
- **브랜드가 아닌 것:** 개인 상호(홍대약국, 마포원조최대포집), 지역명+업종명(강남미용실, 역삼세탁소)
- 판단이 어려우면 null로 반환한다

**브랜드명 정규화:**
- 같은 브랜드의 다른 표기는 가장 널리 알려진 공식 명칭 하나로 통일한다
- CU/씨유 → "CU", 스타벅스/STARBUCKS → "스타벅스", GS25/지에스25 → "GS25"
- 한국에서 더 널리 쓰이는 표기를 우선한다
- 브랜드명은 한국에서 통용되는 표기로 반환한다. 영문 브랜드는 원어 표기를 유지한다 (예: CU, GS25, ZARA)

**배달앱 규칙:**
- "배민1 홍대치킨", "쿠팡이츠 OO점" 등 배달앱 경유 결제는 배달앱 자체를 브랜드로 인식한다
- 브랜드: "배달의민족", "쿠팡이츠", "요기요" 등

**백화점 규칙 (카드 정보 연동):**
- **식품관 구매:** 매출처명에 백화점 키워드가 포함되어 있고, 식품 관련 키워드(과일, 정육, 수산, 야채, 반찬, 베이커리 등)도 포함되면 → 브랜드: "현대백화점 식품관" (또는 해당 백화점명), 카테고리: 식비
- **입점 브랜드 구매:** 매출처명에서 브랜드를 인식할 수 있으면(나이키, 자라, 유니클로 등) → 해당 브랜드로 분류. 백화점은 결제 장소일 뿐이다
- **브랜드 인식 불가:** 매출처명으로 브랜드를 판단할 수 없으면 → 브랜드 null, 카테고리만 분류
- **카드 정보는 참고용 힌트로만 사용한다.** "현대백화점카드로 결제 = 현대백화점에서 결제"라는 추론을 하지 않는다. 현대백화점카드는 어디서든 쓸 수 있다. 매출처명 자체에 백화점 키워드가 있을 때만 백화점 컨텍스트를 적용한다

**해외 매출처 처리:**
- SMS에 붙는 통화 표시(달러/유로/엔화 등)를 무시하고 브랜드명을 판단한다
- 공항 면세점(DFS, Lotte Duty Free 등), 해외 드럭스토어(Boots, dm 등), 해외 마트(Tesco, Carrefour 등), 해외 의류 브랜드, 해외 백화점도 브랜드로 인식한다

## 4. 대시보드 표시 규칙

**브랜드 매출처:** 같은 브랜드끼리 합산하여 개별 표시. "CU 5건 3만원". 클릭하면 지점별 개별 내역이 나온다.

### 비브랜드 매출처 표시 기준 — 연간
- 2건 이상이면 개별 표시 (자주 가는 곳)
- 1건이어도 10만원 초과면 개별 표시 (큰 지출)
- 1건이고 10만원 이하면 카테고리별 기타로 묶기

### 비브랜드 매출처 표시 기준 — 월간
- 기타 묶기 없이 전부 개별 표시
- 브랜드 묶기가 적용되면 월간 항목 수가 충분히 줄어드므로 기타 묶기를 적용하지 않는다
- 랭킹 상위 N개만 보여주고 나머지는 "더 보기"로 접어두기
- 추후 잡음이 느껴지면 월간 기타 묶기를 추가 검토
- 코드 구조상 `getMerchantBreakdown`에 기타 묶기 옵션 파라미터를 설계해두어, 나중에 월간에도 쉽게 켤 수 있도록 한다

### 카테고리별 기타 표시
"외식 기타 8건 12만원", "의료 기타 5건 4만원"처럼 카테고리 아이콘과 함께 표시. 클릭하면 해당 카테고리 내 상호별 소계 팝업.

### 랭킹 정렬 규칙
브랜드 항목과 카테고리별 기타 항목을 금액 내림차순으로 통합 정렬한다. 두 종류가 금액 기준으로 자연스럽게 섞인다.

## 5. 카테고리 아이콘

브랜드 아이콘이 없는 매출처는 카테고리 아이콘으로 표시한다. 현재 기본 고양이 아이콘 대신 12개 카테고리별 아이콘을 사용한다. 아이콘 이미지는 사용자가 별도 제공.

## 6. 입력 폼 변경

**현재 폼:** 매출처, 금액, 카테고리, 카드, 별명, 아이콘 URL

### 변경 후 폼 — 브랜드 매출처 (brand가 있을 때)
- 매출처, 금액, 카테고리, 카드
- 브랜드 (읽기 전용 + 수정 버튼)
- 아이콘 URL — 브랜드에 연결된다 (brandIcons에 저장)

### 변경 후 폼 — 비브랜드 매출처 (brand가 null일 때)
- 매출처, 금액, 카테고리, 카드
- 브랜드 필드 숨김 (비브랜드 매출처에 수동으로 브랜드를 입력할 수 없다)
- 아이콘 URL — 이 매출처 개별로 저장된다 (merchantIcons에 저장)

**비브랜드 매출처에 브랜드를 입력할 수 없는 이유:** 비브랜드 매출처는 다른 지점이 없을 확률이 높아 묶을 대상이 없다. 브랜드 수동 입력을 허용하면 별명 시스템("외식", "약국" 같은 카테고리성 묶기)이 재발한다. 비브랜드 매출처를 시각적으로 구분하고 싶으면 아이콘만 설정한다.

## 7. 보호 규칙

### 아이콘 변경
- **브랜드 매출처:** "이 브랜드 전체 N건에 새 아이콘이 적용됩니다. 계속할까요?"
- **비브랜드 매출처:** 해당 매출처에만 적용되므로 확인 없이 저장한다.

### 브랜드 변경
수정 버튼 클릭 후 다른 브랜드로 바꾸면, "이 항목만 변경" 또는 "이 브랜드 전체 변경" 선택지를 제공한다.

**"이 항목만 변경":**
- 이 expense의 brand 필드만 변경한다
- 해당 매출처명에 대해 `brandOverrides`에 기록하여 미래에 같은 매출처명이 들어오면 새 브랜드를 적용한다

**"이 브랜드 전체 변경":**
- 같은 brand 값을 가진 모든 expense의 brand를 변경한다 (브랜드 단위 일괄 변경)
- `brandOverrides`에는 기록하지 않는다 — 이미 DB를 직접 업데이트했고, 미래 SMS는 Gemini가 독립적으로 판단한다
- **주의:** 전체 변경은 과거 데이터만 일괄 수정하며, 미래 SMS는 Gemini가 독립적으로 판단한다. Gemini가 동일 매출처를 원래 브랜드로 다시 분류할 수 있으며, 이는 의도된 동작이다. 미래 SMS에도 변경을 강제하려면 "이 항목만 변경"을 개별 매출처명마다 적용해야 한다.
- 새 브랜드에 이미 아이콘이 있으면 "○○ 아이콘을 적용할까요?" 확인

**소급 반영:** 두 경우 모두 이미 집계된 월간/연간 통계에 소급 반영된다 (의도된 동작: 사용자가 잘못된 분류를 고치는 것이므로 과거도 바로잡는다).

### 매출처명 변경
"매출처명이 변경되면 기존 브랜드 매칭이 끊어질 수 있습니다. 계속할까요?" (이미 구현됨)

### brand를 null로 변경
Gemini가 브랜드를 잘못 잡았을 때(개인 상호를 프랜차이즈로 인식 등), 사용자가 브랜드를 삭제할 수 있어야 한다. brand 필드를 비우면 null로 저장하고, brandOverrides에 `{brand: null}`을 기록하여 "이 매출처는 비브랜드"라는 명시적 지정으로 취급한다.

## 8. 기존 데이터 처리 (한 번만)

시스템 전환 시 기존 expense 전체에 브랜드 정보가 없으므로 일괄 처리가 필요하다.

### 1단계 — 매핑 테이블 생성
- Gemini에게 기존 매출처 전체(고유 매출처명 목록)를 보내서 브랜드명을 받아온다
- `reclassifyAllExpenses()` 방식과 동일하게 일괄 실행
- 결과를 매출처명 → 브랜드 매핑 테이블로 저장
- 기존 별명 데이터를 참고하여 매칭 검증 (별명 "CU"가 있으면 브랜드 "CU"와 일치하는지)
- **검증:** 같은 매출처명에 서로 다른 브랜드가 할당된 건이 있으면 경고 로그 출력
- **유사 브랜드명 검증:** 같은 프랜차이즈인데 표기가 다른 브랜드명이 있으면 경고 로그 출력 (예: "CU"와 "씨유"가 동시에 존재)

### 2단계 — 일괄 적용
- 사용자가 매핑 테이블을 확인한 뒤, 전체 expense에 brand 필드를 일괄 적용
- 완료 후 기존 별명 시스템은 비활성화하되 데이터는 보존 (롤백용)

### merchantIcons → brandIcons 마이그레이션
- 1단계 일괄 분류로 생성된 `매출처명 → 브랜드` 매핑을 활용하여, merchantIcons의 키를 브랜드명으로 변환한다
- 같은 브랜드에 여러 아이콘이 매핑되면 가장 마지막으로 설정된 것을 사용한다
- 변환할 수 없는 키(비브랜드 매출처의 아이콘)는 merchantIcons에 그대로 유지한다 (비브랜드 아이콘 조회용)
- 마이그레이션 스크립트는 8번 2단계 처리에 포함한다

## 9. 데이터 스키마 변경

### expense 항목에 brand 필드 추가
```
{ id, amount, category, merchant, card, memo, date, time, created, source, brand }
```

### 브랜드 아이콘 저장소 (K.brandIcons, 새로 추가)
```
{ "CU": "https://cu-logo.png", "스타벅스": "https://starbucks-logo.png", ... }
```

1단계에서 `K.brandIcons` 키를 빈 객체로 생성한다. 기존 `K.merchantIcons`는 건드리지 않는다. Gemini 일괄 분류가 끝나서 매출처명 → 브랜드 매핑이 확정된 후, 기존 merchantIcons에서 마이그레이션을 수행한다 (8번 참조).

### 기존 merchantIcons 유지
비브랜드 매출처의 개별 아이콘 저장소로 계속 사용한다. 브랜드 시스템 완성 후에도 제거하지 않는다.

### 매출처명 → 브랜드 오버라이드 (K.brandOverrides, 새로 추가)
```
{ "씨유홍대3호점": { "brand": "CU", "created": "2026-03-13" }, ... }
```
사용자가 수동으로 브랜드를 수정하고 "이 항목만 변경"을 선택했을 때 기록된다. 새 SMS가 같은 매출처명으로 들어오면 Gemini 결과보다 이 오버라이드를 우선 적용한다. `created` 필드는 향후 오래된 오버라이드의 Gemini 재판단 허용 옵션을 위해 기록해둔다 (현재는 사용하지 않음).

brand 값이 null인 오버라이드는 "이 매출처는 비브랜드"라는 명시적 지정으로 취급한다.

**"이 브랜드 전체 변경"을 선택한 경우:** brandOverrides에 기록하지 않는다. 이미 DB의 모든 해당 expense를 직접 업데이트했고, 미래 SMS는 Gemini가 판단한다 (7번 참조).

### 기존 merchantAliases
당분간 유지, brand 필드가 안정화되면 제거.

### 아이콘 조회 순서
1. brand가 있으면 → `K.brandIcons[brand]`
2. brand가 null이면 → `K.merchantIcons[merchant]`
3. 둘 다 없으면 → `MERCHANT_LOGOS` 매핑 (파비콘 도메인 검색, 기존 로직)
4. 모두 없으면 → 카테고리 아이콘

**MERCHANT_LOGOS 처리:** 기존 MERCHANT_LOGOS 매핑(키워드 → 파비콘 도메인)은 brandIcons/merchantIcons에 아이콘이 없을 때의 폴백으로 유지한다. brandIcons 커버리지가 충분해지면 단계적으로 폐기를 검토한다.

### 동기화 대상 추가
`K.brandIcons`와 `K.brandOverrides`를 `saveDatabase`/`loadDatabase` 대상에 포함한다. 현재 `saveDatabase`는 `app_database.json` 전체를 덮어쓰므로 키만 추가하면 저장은 자동 포함되지만, `loadDatabase`에서 이 키들을 LocalStorage에 복원하는 코드를 명시적으로 추가해야 한다.

## 10. 작업 순서

**멀티유저:** 두 사용자(leftjap, soyoun312) 모두 적용. 코드 변경은 공통이고, 기존 데이터 일괄 처리(8번)는 각 사용자의 DB에 대해 별도 실행한다.

---

### 1단계 — 데이터 기반 (스키마 + 저장소)

- expense 스키마에 brand 필드 추가
- `K.brandIcons` 키 추가 (빈 객체로 시작), storage.js/data.js에 CRUD 추가
- `K.brandOverrides` 키 추가 (빈 객체, 값은 `{brand, created}` 형태), data.js에 CRUD 추가
- sync 대상에 brandIcons, brandOverrides 추가 (loadDatabase에서 LocalStorage 복원 코드 포함)
- 이 단계까지는 기존 동작에 영향 없음 (brand가 undefined여도 기존 코드가 깨지지 않음)

**1단계 검증:**
- 기존 기능 회귀 테스트
- brand 필드가 없는(undefined) 기존 데이터로 대시보드/폼/통계/동기화가 정상 동작하는지 확인
- `getMerchantBreakdown`의 groupBy 기본값이 `'alias'`로 유지되는지 확인
- `getMerchantIconHtml`이 brand가 undefined일 때 기존 경로(merchantIcons → MERCHANT_LOGOS)를 타는지 확인

---

### 1-2단계 — Gemini 연동

- Code.gs의 `classifyMerchantWithGemini` 인터페이스 변경: `(merchant, config)` → `(merchant, card, config)`
- Gemini 프롬프트에 브랜드명 질의 추가 (3번 가이드라인 전체 포함: 브랜드 기준, 브랜드명 정규화, 응답 언어, 배달앱 규칙, 백화점 규칙, 해외 매출처, 카드 정보 힌트 제한)
- Gemini 응답 형식을 JSON으로 변경 + JSON 파싱 실패 시 폴백 로직 추가
- `saveExpenseFromSMS()`에서 brand 저장 + brandOverrides 우선 적용 (category는 Gemini 결과 그대로 사용, brandOverrides 체크는 최종 brand 결정 시점에서 수행 — 3번 참조)
- `autoMatchCategoryServer` 폴백 시 brand는 항상 null (단, 이후 brandOverrides 체크는 동일하게 수행)
- 이 시점부터 새 SMS에 brand가 붙기 시작함

---

### 1-3단계 — 기존 데이터 일괄 브랜드 분류

- 1-2가 검증된 후 실행
- 기존 expense 전체에 brand 일괄 적용 (8번 처리)
- merchantIcons → brandIcons 마이그레이션 (8번 참조)
- 비브랜드 매출처의 아이콘은 merchantIcons에 그대로 유지
- **brandOverrides 보호:** `reclassifyAllExpenses` 실행 시, brandOverrides에 이미 사용자가 수동 지정한 매출처명은 건너뛴다. 사용자가 수동으로 고친 브랜드가 Gemini 재분류로 덮어씌워지면 안 된다.

### 2단계 — 대시보드 표시

**2단계 전환 조건:** 사용자가 수동으로 전환한다. 1-3 일괄 분류 완료 후 사용자가 확인하면 `'brand'` 모드로 전환.
